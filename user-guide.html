Phase20 working guide · Design to development

From feature idea to developer handoff

Use this workflow to turn a learner or teacher need into an interactive prototype, a clear Notion feature specification, and implementation-ready tasks.

One feature record, several views

The Notion Feature & Design Tracker is both the detailed specification and pipeline source of truth. Its Pipeline, Features, and Ready for handoff views show the same records for different needs. This repository holds interactive prototypes. Keep the feature name, status, description, links, and decisions consistent between them.

The repeatable workflow

Five stages from idea to build

  1. 1

    Start with the feature context

    Find or create the matching record in Feature & Design Tracker under Design & Feature Ideas. Read its documentation before making assumptions.

  2. 2

    Explore the experience

    Choose one primary product-area folder, copy the reusable template when useful, and build the key learner or teacher journey as an interactive prototype.

  3. 3

    Make it ready for feedback

    Test the main flow, keyboard access, focus states, mobile layout, reduced motion, relative links, and basic HTML and JavaScript integrity.

  4. 4

    Document the behaviour in Notion

    Update that same tracker record with the user journey, screen states, requirements, edge cases, accessibility expectations, prototype links, and testable acceptance criteria. The record will appear automatically in the appropriate views.

  5. 5

    Prepare the developer handoff

    When the feature is approved, create a linked implementation plan and focused developer tasks. Keep unfinished ideas and open questions visible instead of hiding them in acceptance criteria.

Notion feature record

What the developer should be able to find

Why and who

  • Feature summary and user need
  • Goals and non-goals
  • Learner or teacher journey
  • Primary action and expected outcome

What it does

  • Screen-by-screen behaviour
  • Functional requirements and content rules
  • Loading, empty, error, and interrupted states
  • Data or permissions needed

How to validate it

  • Keyboard and screen-reader expectations
  • Responsive and reduced-motion behaviour
  • Testable acceptance criteria
  • Decisions, assumptions, and open questions

Always include

Links and status

  • Live GitHub Pages prototype URL
  • Repository file path and product area
  • Matching Notion tracker entry
  • Current standard status: Planned, Early concept, In progress, Ready for feedback, or Archived
  • Short changelog showing what changed and when

A simple request to use with Codex

“Document the [feature name] prototype in Notion for developer handoff.”

Codex should inspect the relevant Notion feature and repository prototype, update the matching page, add the live URL, and report any missing decisions or open questions without inventing requirements.
